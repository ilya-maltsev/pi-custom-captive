# pi-custom-captive

Минималистичный портал самообслуживания для [privacyIDEA](https://www.privacyidea.org/).

Узкий прокси к REST API privacyIDEA ровно с двумя точками входа и намеренно минимальной поверхностью привилегий:

- **Пользовательский поток** — однократная самостоятельная регистрация TOTP. После регистрации пользователь блокируется до тех пор, пока администратор не удалит его токен.
- **Административный поток** — двухэтапный вход challenge-response (пароль → OTP) полностью через endpoint `/auth` PI с использованием `transaction_id`. После аутентификации с 2FA администраторы попадают на таблицу TOTP-токенов всего realm (с сортировкой, фильтрацией по столбцам) с действиями включения / отключения / удаления. Учётные данные никогда не покидают память сервера — пароль **никогда** не сохраняется в сессии и не передаётся обратно в браузер.

Портал **не имеет базы данных** и **не использует сервисный аккаунт**. Каждый вызов PI выполняется с JWT актора — PI автоматически ограничивает область видимости по JWT вызывающего. События передаются через syslog и могут быть перенаправлены в rsyslog-контейнер родительского проекта `privacyidea-docker`.

## Архитектура

```
Browser ──> [ pi-custom-captive :6443 ] ── PI REST API ──> [ reverse_proxy :8443 ] ──> [ privacyidea :8080 ]
```

| Компонент | Роль |
|-----------|------|
| **app** (Django) | отображает портал, вызывает PI REST API с JWT актора |
| **reverse_proxy** (nginx) | терминация TLS |

Портал никогда не действует от имени пользователя с привилегиями выше, чем у самого пользователя. JWT пользователя выполняет пользовательскую работу; JWT администратора — административную.

Вызываются только следующие endpoint-ы PI:

- `/auth` — проверка пароля и OTP для **обоих** шагов 2FA администратора (шаг 1: пароль → challenge, шаг 2: `transaction_id` + OTP → JWT) и аутентификация пользователя по паролю. Возвращает JWT, который портал использует для остальной части сессии.
- `/validate/check` — используется только в пользовательском потоке для верификации только что зарегистрированного TOTP по сгенерированному секрету. JWT не требуется.
- `/token/` — список токенов, видимых вызывающему по JWT. Главная страница администратора перечисляет все TOTP-токены в управляемом realm через `realm=<CAPTIVE_PI_REALM>&type=totp`.
- `/token/init` — регистрация TOTP. При вызове с JWT пользователя регистрирует для этого пользователя; при вызове с JWT администратора может указать любого пользователя через `user=`.
- `/token/<serial>` DELETE, `/token/enable`, `/token/disable`, `/token/reset` — административные мутации; выполняются с JWT администратора (полученным после полной 2FA при входе).

## Требования к политикам PI

Для административного входа с challenge-response требуется политика privacyIDEA с настройками:

| Параметр | Значение |
|----------|----------|
| **Scope** | `authentication` |
| `challenge_response` | `totp` |
| `otppin` | `userstore` |
| `passOnNoToken` | `true` |

Это заставляет PI:
1. Принимать пароль AD/LDAP в качестве PIN (`otppin: userstore`).
2. Запускать TOTP challenge при получении валидного пароля для пользователя, у которого есть TOTP-токен (`challenge_response: totp`).
3. Разрешать пользователям без токенов аутентифицироваться только по паролю (`passOnNoToken: true`) — используется для первоначальной регистрации администраторов.

## Рабочие процессы

### Пользовательский поток — однократная самостоятельная регистрация

```
[  Browser  ]                [  captive portal  ]                    [  privacyIDEA  ]
     |                               |                                       |
     | GET /                         |                                       |
     |------------------------------>|                                       |
     |<------------- 200 sign-in form|                                       |
     |                               |                                       |
     | POST / user,password          |                                       |
     |------------------------------>|                                       |
     |                               | POST /auth user,password,realm        |
     |                               |-------------------------------------->|
     |                               |<- 200 JWT (role=user, rights=[..,     |
     |                               |             enrollTOTP, ..])          |
     |                               | store JWT in signed-cookie session    |
     |                               |                                       |
     |                               | GET /token/?type=totp&active=true     |
     |                               |     (auth: user JWT, no user= param)  |
     |                               |-------------------------------------->|
     |                               |<- 200 {tokens: []}                    |
     |                               |                                       |
     |<- 302 /enroll/                |                                       |
     | GET /enroll/                  |                                       |
     |------------------------------>|                                       |
     |                               | POST /token/init                      |
     |                               |     type=totp, genkey=1,              |
     |                               |     hashlib=sha256, otplen=6,         |
     |                               |     (auth: user JWT, no user= param)  |
     |                               |-------------------------------------->|
     |                               |<- 200 {serial: TOTP0002702F,          |
     |                               |         googleurl.value: otpauth://…} |
     |<- 200 QR page                 |                                       |
     |    [ user scans in Google     |                                       |
     |      Authenticator ]          |                                       |
     |                               |                                       |
     | POST /enroll/ otp=123456      |                                       |
     |------------------------------>|                                       |
     |                               | POST /validate/check user,pass=otp,   |
     |                               |     realm  (no JWT)                   |
     |                               |-------------------------------------->|
     |                               |<- 200 authentication=ACCEPT           |
     |                               |                                       |
     |                               | session.flush()                       |
     |<- 302 /done/                  |                                       |
```

Любой последующий `GET /` для того же пользователя повторяет шаг `GET /token/?type=totp&active=true`; на этот раз `count=1`, поэтому портал перенаправляет на страницу **Уже зарегистрирован**. Это и есть блокировка: она сохраняется до тех пор, пока администратор не удалит токен.

### Административный поток — вход с challenge-response (без учётных данных в сессии)

```
[  Browser  ]                [  captive portal  ]                    [  privacyIDEA  ]
     |                               |                                       |
     |  ─── Шаг 1: запуск challenge ───                                     |
     |                               |                                       |
     | POST /admin/login/            |                                       |
     |  user, password               |                                       |
     |------------------------------>|                                       |
     |                               | POST /auth                            |
     |                               |  username=<admin>, password=<pw>      |
     |                               |-------------------------------------->|
     |                               |<- 200 {result.status: true,           |
     |                               |   result.value: false,                |
     |                               |   detail.transaction_id: "036300…"}   |
     |                               |                                       |
     |                               | discard password (GC'd after response)|
     |<- 200 OTP form                |                                       |
     |    [username, transaction_id: |                                       |
     |     hidden]                   |                                       |
     |                               |                                       |
     |  ─── Шаг 2: завершение аутентификации ───                             |
     |                               |                                       |
     | POST /admin/login/            |                                       |
     |  username, transaction_id, otp|                                       |
     |------------------------------>|                                       |
     |                               | POST /auth                            |
     |                               |  username=<admin>,                    |
     |                               |  transaction_id=<tid>,                |
     |                               |  password=<otp>                       |
     |                               |-------------------------------------->|
     |                               |<- 200 JWT (role=admin)                |
     |                               |                                       |
     |                               | session: admin_token=JWT              |
     |<- 302 /admin/                 |                                       |
     |                               |                                       |
     |  ─── Полный доступ (2FA завершена) ──                                 |
     |                               |                                       |
     | GET /admin/                   |                                       |
     |------------------------------>|                                       |
     |                               | GET /token/?realm=<realm>&type=totp   |
     |                               |-------------------------------------->|
     |                               |<- 200 {tokens: [...]}                 |
     |<- 200 realm-wide TOTP table   |                                       |
     |    (sort/filter, per-row      |                                       |
     |     enable/disable/delete)    |                                       |
     |                               |                                       |
     | POST /admin/token/<s>/delete/ |                                       |
     |------------------------------>|                                       |
     |                               | DELETE /token/<serial>                |
     |                               |     (auth: admin JWT)                 |
     |                               |-------------------------------------->|
     |                               |<- 200                                 |
     |<- 302 /admin/                 |                                       |
```

Ключевые свойства этого потока:
- **Пароль никогда не сохраняется — нигде.** Пароль шага 1 используется однократно для `POST /auth`, затем очищается сборщиком мусора. Он не хранится в сессии, не шифруется в cookie и не отправляется обратно из браузера. Только `transaction_id` передаётся туда и обратно как скрытое поле формы.
- **Оба шага используют один endpoint (`/auth`).** Старая реализация смешивала `/validate/check` (шаг 1) с `/auth` (шаг 2) и требовала вспомогательный модуль для шифрования пароля между ними; этот модуль (и его симметричный ключ) были удалены.
- **Дополнительная верификация не требуется.** 2FA завершается во время входа — все действия по управлению доступны сразу.
- **Неверные пароли обнаруживаются на шаге 1.** `POST /auth` выбрасывает `PIClientError` при невалидном пароле; ответ challenge-response (`status=true`, `value=false`, `transaction_id` в `detail`) означает, что пароль принят и требуется OTP.

Администраторы без зарегистрированного TOTP проходят через `passOnNoToken` и сразу перенаправляются на страницу регистрации. После регистрации им необходимо повторно войти — следующий вход запускает поток challenge-response.

### Поток mTLS

Аутентификация по заголовку через клиентский сертификат, верифицированный реверс-прокси, реализована в настройках/middleware, но **в настоящее время отключена в коде**, поскольку mTLS не создаёт PI JWT, а портал больше не использует сервисный аккаунт. Этот путь кода возвращает 501 с объяснением ситуации. Для повторного включения необходимо:

- использование политики PI passthru-by-certificate, чтобы `/auth` или `/validate/check` принимали сертификат как фактор, или
- повторное введение сервисного аккаунта для ветки mTLS-only с явным флагом конфигурации.

Оба варианта остаются открытыми; переменные `MTLS_*` сохранены, чтобы каркас был готов при необходимости.

## Конфигурация

Полный список переменных см. в `environment/application-captive.env`. Ключевые переменные:

| Переменная | По умолчанию | Описание |
|------------|-------------|----------|
| `PI_API_URL` | `https://host.docker.internal:8443` | базовый URL REST API privacyIDEA |
| `PI_REALM` | `defrealm` | устаревшая переменная realm — используется, когда `CAPTIVE_PI_REALM` не задан |
| `CAPTIVE_PI_REALM` | *(пусто → fallback на `PI_REALM`)* | realm, в котором работает портал (регистрация пользователей + таблица токенов администратора) |
| `CAPTIVE_ADMIN_PREFIX` | `admin` | сегмент URL для административных endpoint-ов. Установите, например, `manage` для `/manage/`, `/manage/login/`, `/manage/token/<s>/delete/`. Без ведущих/завершающих слешей. |
| `DJANGO_LANGUAGE_CODE` | `en` | язык интерфейса по умолчанию, когда у посетителя нет выбора в сессии/cookie и нет подходящего заголовка `Accept-Language`. Допустимые значения: `en`, `ru`. |
| `PROXY_PORT` | `6443` | внешний HTTPS-порт |
| `OTPAUTH_ISSUER` | `privacyIDEA` | переопределяет строку издателя в приложении-аутентификаторе (например, `VPN-GATE1`) — см. ниже |
| `OTPAUTH_LABEL_ATTR` | `username` | атрибут пользователя PI, отображаемый как строка аккаунта в приложении-аутентификаторе |
| `TOKEN_SERIAL_PREFIX` | *(пусто)* | если задано, портал формирует пользовательские серийные номера PI — см. ниже |
| `TOKEN_SERIAL_SUFFIX` | `username` | атрибут пользователя PI для средней части пользовательского серийного номера |
| `SYSLOG_*` |  | опциональная удалённая пересылка syslog (та же схема, что и у pi-vpn-pooler) |
| `MTLS_*` |  | аутентификация по заголовку mTLS (каркас присутствует; поток в настоящее время возвращает 501, см. выше) |

Намеренно **отсутствуют** `PI_SERVICE_USER` / `PI_SERVICE_PASSWORD`. Портал никогда не аутентифицируется ни от чьего имени, кроме текущего актора.

### Отображение в приложении-аутентификаторе и серийный номер токена

Две независимые настройки:

**1. Что пользователь видит в Google Authenticator / Authy / 2FAS.** URI otpauth, который портал кодирует в QR, управляет строкой аккаунта. По умолчанию PI использует `privacyIDEA: TOTP0002702F`, что бесполезно для конечных пользователей. Портал переписывает URI на лету:

- `OTPAUTH_ISSUER` (по умолчанию `privacyIDEA`) → отображается как жирная строка в списке приложения.
- `OTPAUTH_LABEL_ATTR` (по умолчанию `username`) → какой атрибут пользователя PI становится строкой аккаунта под ним. `username` не требует дополнительных вызовов PI; другие значения (`email`, `givenname`, `mobile`, `custom_*`) запускают `GET /user/` с JWT пользователя — см. примечание о политике ниже.

Пример: `OTPAUTH_ISSUER=VPN-GATE1 OTPAUTH_LABEL_ATTR=username` заставляет приложение-аутентификатор показывать **VPN-GATE1 · LatrStr**.

Экран регистрации также показывает необработанный секрет в base32 в блоке с копированием по клику (сгруппированный `ABCD EFGH IJKL …`) для пользователей, чьи устройства не могут сканировать QR — больше никаких скрытых «Показать URI настройки».

**2. Что PI хранит как серийный номер токена.** По умолчанию PI назначает `TOTPXXXXXXXX`, что непрозрачно для администраторов. Когда `TOKEN_SERIAL_PREFIX` задан, портал передаёт предварительно сформированный `serial=` в `/token/init` в формате:

```
{TOKEN_SERIAL_PREFIX}-{SANITIZED(user.<TOKEN_SERIAL_SUFFIX>)}-{SHORT_HASH}
```

- `SANITIZED(…)` оставляет только `[A-Z0-9]` и приводит к верхнему регистру.
- `SHORT_HASH` — 6 случайных hex-символов — предотвращает коллизии при санитизации (например, `a.b` и `ab` оба сворачиваются в `AB`) и позволяет различать повторные регистрации в журнале аудита PI.
- Если `TOKEN_SERIAL_SUFFIX=""`, средний сегмент опускается: `{PREFIX}-{HASH}`.

Пример с `TOKEN_SERIAL_PREFIX=VPN-GATE1 TOKEN_SERIAL_SUFFIX=username` и пользователем `SrinPur`:

```
VPN-GATE1-SRINPUR-8240DC
```

Администраторы, просматривая PI, теперь могут группировать токены по префиксу (`VPN-GATE1-*`) и сразу видеть, кому принадлежит каждый из них.

> [!NOTE]
> **Требование к политике PI для атрибутов, отличных от username.** Получение `email` / `givenname` / `mobile` через `/user/?username=<self>` требует, чтобы политика PI пользовательской области предоставляла действие `userlist`. Dev-seed в `privacyidea-docker` не предоставляет его, поэтому атрибуты, отличные от username, молча откатываются к логину — портал логирует `attr=<X> not found on user=<U>; using fallback`, и регистрация всё равно проходит успешно. В production добавьте политику authentication-scope или user-scope с `action=userlist`, ограниченную realm, если хотите использовать эти атрибуты.

## Интернационализация (i18n)

Портал поддерживает русский и английский языки. Язык по умолчанию задаётся переменной окружения `DJANGO_LANGUAGE_CODE` (по умолчанию `en`). Пользователи могут переключать язык через кнопки в верхней панели.

Файлы перевода находятся в `locale/ru/LC_MESSAGES/django.po`. После редактирования файла `.po` перекомпилируйте:

```bash
msgfmt -o locale/ru/LC_MESSAGES/django.mo locale/ru/LC_MESSAGES/django.po
```

При запуске в Docker пересоберите образ, чтобы файл `.mo` был включён.

## Быстрый старт

```bash
make secrets                       # генерация DJANGO_SECRET_KEY
# отредактируйте environment/application-captive.env
make cert                          # самоподписанный TLS для nginx реверс-прокси
make stack                         # запуск production compose
```

Откройте `https://localhost:6443/` для пользовательского портала, `https://localhost:6443/admin/login/` для административной области (или `/<CAPTIVE_ADMIN_PREFIX>/login/`, если вы изменили префикс).

Для разработки: `make dev` (запуск на `http://localhost:6000`, hot-reload, без TLS).

При запуске как часть родительского fullstack `privacyidea-docker` контейнер captive доступен по `https://localhost:6443/` через центральный nginx реверс-прокси (`:445 → captive:8000`).

## Опционально: аутентификация по заголовку mTLS (каркас)

Портал может пропустить шаг ввода пароля пользователя и определить идентичность из клиентского TLS-сертификата, верифицированного вышестоящим nginx. Примеры конфигурации nginx находятся в `templates/nginx-mtls.{http,server}.example.conf`. Подробности о том, почему этот поток в настоящее время возвращает 501, см. в разделе «Рабочие процессы» выше — каркас на месте; связывание идентичности с реальной PI-сессией — это недостающая часть.

### Переменные окружения

| Переменная | По умолчанию | Описание |
|------------|-------------|----------|
| `MTLS_ENABLED` | `false` | Основной переключатель для пользовательского потока. Административный поток не затрагивается. |
| `MTLS_USER_HEADER` | `HTTP_X_SSL_USER` | Django META ключ, содержащий имя пользователя. |
| `MTLS_VERIFY_HEADER` | `HTTP_X_SSL_VERIFY` | Django META ключ, содержащий `$ssl_client_verify` nginx. |
| `MTLS_REQUIRED_VERIFY_VALUE` | `SUCCESS` | Значение, которое должен иметь заголовок верификации для пропуска запроса. |

### OCSP (отзыв клиентских сертификатов)

nginx ≥ 1.19 поддерживает OCSP-верификацию **клиентских сертификатов** через `ssl_ocsp on;` (не путать с OCSP stapling для серверного сертификата). URL ответчика берётся из расширения AIA каждого сертификата по умолчанию — переопределяйте через `ssl_ocsp_responder` только если ваш CA не заполняет AIA. Кэшируйте ответы через `ssl_ocsp_cache`. Если ответчик недоступен, nginx отклоняет соединение по умолчанию; для отзыва на основе CRL используйте `ssl_crl /etc/nginx/ssl/user-ca.crl;`.

### Замечания по безопасности

- Не выставляйте gunicorn (`:8000`) напрямую, когда `MTLS_ENABLED=true`. Всегда используйте nginx в качестве фронтенда.
- Пример конфигурации начинается с `proxy_set_header X-SSL-User "";` для обнуления заголовка, который может быть подставлен атакующим, перед перезаписью верифицированным значением nginx. Сохраняйте эту строку.
- Ограничьте доверенных издателей в `ssl_client_certificate` — не используйте системный набор корневых сертификатов.
- Административный поток никогда не зависит от `MTLS_ENABLED`. Администраторы по-прежнему входят по паролю и проходят 2FA через challenge-response.
